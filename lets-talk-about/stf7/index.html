STF-7: The Silly 7-bit Transformation Format

What follows is a definition for the STF-7 encoding scheme I have just created.

Features

This encoding has a number of advantages over UTF-7:

  • Self-synchronization.
  • Native encoding of supplementary characters (without surrogates).
  • Direct and indirect characters occupy different ASCII ranges.
  • U+10FFFF encodes in six ASCII values.

Encoding Steps

Some Unicode characters are encoded directly as single ASCII values. Others are encoded indirectly as multiple values, up to and including six.

Direct Encoding

The following character ranges are encoded directly by making the encoded ASCII value the same as the Unicode scalar value.

Low High Description
U+00 U+20 Control codes and space
U+30 U+39 Numbers
U+41 U+5A Latin uppercase
U+61 U+7A Latin lowercase

Directly encoded characters can be interpreted as-is by an ASCII system without STF-7 support.

Indirect Encoding

Any Unicode character not directly encoded is indirectly encoded. In this case, the scalar value is split into chunks of four bits.

Low High ASCII Values
U+21 U+2F 2
U+3A U+40 2
U+5B U+60 2
U+7B U+FF 2
U+100 U+FFF 3
U+1000 U+FFFF 4
U+10000 U+FFFFF 5
U+100000 U+10FFFF 6

The number of ASCII values needed will reflect the number of four bit chunks the scalar value will be divided into. Padding in the most significant chunk is done by prepending the bits there with zeros.

Initial Chunk

Start with the most significant chunk. Its value will inherently be between 0x0 and 0xF. Output a single ASCII value in place of it as specified below:

After this, the next Unicode value can be encoded.

Decoding Considerations

The decoding steps are performed in reverse. A main consideration to keep in mind is that Unicode values that can be directly encoded should not be decoded indirectly. In other words, anything that can be directly encoded that isn't directly encoded should be considered an encoding error and rejected.
